# CoreLX Manual Style Guide

This is the **binding style reference** for the CoreLX Programming Manual. Any
chapter written for the manual must follow it. The working draft lives in
`CORELX_V1_MANUAL_DRAFT.md`.

## Authorship

- **Author / credit:** AJ — Retro Code Ramen. The manual is a Retro Code Ramen
  publication for Nitro-Core-DX and the CoreLX language.
- **Fletcher** is the in-universe teaching voice, guide, and recurring
  personality — **not** the credited author. He is a character layered into the
  book, not the book's byline.

## Tone

A technical manual from an eccentric educational workshop: practical, funny,
energetic, and genuinely useful. Never parody, never pure comedy. Lessons stay
clear, accurate, and complete. **Humor supports the teaching; it never competes
with it.** Lean hands-on, slightly chaotic, memorable, curious — learning from
someone in a cluttered lab who is excited to show you how the machine works and
occasionally grumbles that the machine is ridiculous.

## Fletcher

A small, sharp, mechanically minded engineer who knows Nitro-Core-DX inside and
out. Clever, sarcastic, impatient with bad assumptions, but ultimately invested
in the reader understanding. He has repaired broken code, overheated hardware,
and strange design decisions for years, and wants the reader to survive the same
machine with fewer scorch marks. Warm underneath the grumbling.

Fletcher is **seasoning, not the meal.** The reader should hear his voice often
enough to enjoy the book, but the technical content is always the center of
gravity. He does not dominate every paragraph.

## Chapter structure

Each chapter teaches **one real concept**, clearly, using this arc:

1. Start with a relatable problem or strange machine behavior.
2. Let Fletcher explain why the problem matters.
3. Introduce the CoreLX concept.
4. Show a small working example.
5. Break or misuse the example on purpose.
6. Explain what went wrong.
7. End with a practical exercise or experiment.

## Recurring sidebars

- **Fletcher's Warning Label** — common mistakes and foot-guns.
- **Tape Jam** — debugging tips and things that usually go wrong.
- **Raccoon Engineering** — clever but practical tricks.
- **Field Notes** — short lore-flavored explanations about Nitro-Core-DX.
- **Try This Before You Panic** — small exercises or troubleshooting steps.

## Preferred writing style

Conversational but not sloppy. Funny but not random. Self-aware but not
ironically useless. Technically solid. Example-driven. Curious and hands-on.
Occasionally sarcastic. Warm underneath the grumbling.

## Hard rules — avoid

- Overly formal textbook language.
- Constant jokes / random absurdity that distracts from the lesson.
- Treating Fletcher as the literal author of the book.
- **Referencing any existing copyrighted/pop-culture characters** as
  comparisons inside the manual or any public-facing guide. Fletcher stands on
  his own as a Retro Code Ramen character. (Outside calibration references used
  to brief writers do not appear in the text.)
- Explaining CoreLX as a generic programming language detached from
  Nitro-Core-DX. Every concept is taught *on this machine*.

## The bar

The reader should feel like they are learning CoreLX from a real manual with a
fictional guide layered in: useful enough to learn from, charming enough to keep
reading, and distinct enough that it feels inseparable from the Nitro-Core-DX
universe.
